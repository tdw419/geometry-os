## /superpowers (Core Workflows)

Enforces the "Red-Green-Refactor" cycle and systematic debugging workflows. Prevents agents from claiming tasks complete until rigorous verification.

### Core Workflow Skills

| Command | Use When | Enforcement |
|---------|----------|-------------|
| `brainstorming` | Starting new feature, need design exploration | Required before new features |
| `test-driven-development` | Implementing with tests first | Hard gate - deletes non-TDD code |
| `systematic-debugging` | Tests failing, need diagnosis | Soft gate - recommends workflow |
| `verification-before-completion` | Before claiming work is done | Hard gate - requires evidence |
| `executing-plans` | Have a written plan to execute | Soft gate |
| `writing-plans` | Need to create implementation plan | Soft gate |
| `receiving-code-review` | Got feedback to address | Soft gate |
| `requesting-code-review` | Need code reviewed | Soft gate |
| `dispatching-parallel-agents` | Have 2+ independent tasks | Soft gate |
| `finishing-a-development-branch` | Implementation complete, need to integrate | Soft gate |
| `subagent-driven-development` | Executing plans with independent tasks | Soft gate |
| `using-git-worktrees` | Starting feature work needing isolation | Soft gate |

### Superpowers Behavior

- **Hard Gates**: Skills that enforce mandatory behavior (e.g., TDD deletes code written before tests)
- **Soft Gates**: Skills that recommend but don't enforce
- **Required Invocation**: `brainstorming` MUST be called before any creative work

    ✦ I'll implement a WGSL Interpreter for our visual SPIR-V binary.                                                                                   
                                                                                                                                                        
                                                                                                                                                        
      Since browsers primarily use WGSL for WebGPU, we can avoid a complex WASM dependency (like Naga) by writing a small Compute Shader (WGSL) that    
    acts as                                                                                                                                             
      the "VM" for our visual instructions. This keeps everything in the browser, matching the current demo.html setup.                                 
                                                                                                                                                        
                                                                                                                                                        
      The Strategy:                                                                                                                                     
       1. `web/executor.wgsl`: A compute shader that loads our .spv binary as a storage buffer and executes it.                                         
       2. `web/SpirvRunner.js`: JavaScript to load the visual program, upload it to the GPU, and trigger the executor.                                  
       3. `web/demo.html`: Update with a "Execute Visual Program" button.                                                                               
                                                                                                                                                        
                                                                                                                                                        
      This approach is highly portable and follows the "The Screen is the Hard Drive" philosophy of Geometry OS—where the code (pixels) is executed     
    directly                                                                                                                                            
      on the visual substrate.                                                                                                                          
                                                                                                                                                        
      ---                                                                                                                                               
                                                                                                                                                        
      Step 1: Create web/executor.wgsl                                                                                                                  
      This shader will decode and execute our visual instructions.                                                                                      
                                                                                                                                                        
                                                                                                                                                        
        1 @group(0) @binding(0) var<storage, read> program: array<u32>;                                                                                 
        2 @group(0) @binding(1) var<storage, read_write> stack: array<f32>;                                                                             
        3 @group(0) @binding(2) var<storage, read_write> output: array<f32>;

        5 @compute @workgroup_size(1)                                                                                                                   
        6 fn main() {                                                                                                                                   
        7     var sp: u32 = 0; // Stack pointer                                                                                                         
        8     var pc: u32 = 5; // Start after SPIR-V header (5 words)                                                                                   
        9                                                                                                                                               
       10     // Simple interpreter for our specific GeoASM subset                                                                                      
       11     loop {                                                                                                                                    
       12         if (pc >= arrayLength(&program)) { break; }                                                                                           
       13                                                                                                                                               
       14         let word = program[pc];                                                                                                               
       15         let opcode = word & 0xFFFFu;                                                                                                          
       16         let count = (word >> 16u) & 0xFFFFu;                                                                                                  
       17                                                                                                                                               
       18         // Handle GeoASM/SPIR-V Opcodes                                                                                                       
       19         if (opcode == 43u) { // OP_CONSTANT                                                                                                   
       20             stack[sp] = bitcast<f32>(program[pc + 2]);                                                                                        
       21             sp = sp + 1;                                                                                                                      
       22         } else if (opcode == 129u) { // OP_FADD                                                                                               
       23             let v2 = stack[sp - 1];                                                                                                           
       24             let v1 = stack[sp - 2];                                                                                                           
       25             stack[sp - 2] = v1 + v2;                                                                                                          
       26             sp = sp - 1;                                                                                                                      
       27         }                                                                                                                                     
       28         // ... more ops ...                                                                                                                   
       29                                                                                                                                               
       30         pc = pc + count;                                                                                                                      
       31     }                                                                                                                                         
       32                                                                                                                                               
       33     output[0] = stack[0]; // Store result                                                                                                     
       34 }

  ❯ # **The Foundations of Morphological GPU Computation: Direct SPIR-V binary Emission from Semantic RGB Glyphs and Hilbert-Sequenced instruction      
    Streams**                                                                                                                                           
                                                                                                                                                        
    The convergence of fractal geometry, visual linguistics, and low-level graphics hardware has enabled a revolutionary compilation path for           
    high-performance computing. At the nexus of this development is the Geometry OS architecture, which utilizes a direct morphological translation     
    from visual patterns to the Standard Portable Intermediate Representation (SPIR-V). By bypassing traditional high-level shading languages such as   
    WebGPU Shading Language (WGSL) or High-Level Shader Language (HLSL), Geometry OS facilitates a more immediate relationship between algorithmic      
    design and execution bytecode.1 This paradigm relies on a dual-layered transformation: first, the mapping of semantic data into the discrete color  
    channels of a digital image, and second, the spatial reordering of these data points through a Hilbert space-filling curve to maintain              
    hardware-level efficiency and locality.3 The result is a system where the structure of the program is intrinsically linked to its visual            
    representation, turning the act of programming into a process of morphological arrangement.                                                         
                                                                                                                                                        
    ## **The Semantic RGB Matrix and Morphological Encodings**                                                                                          
                                                                                                                                                        
    The architectural integrity of the Geometry OS compilation pipeline is predicated on the repurposing of the Red, Green, and Blue (RGB) color        
    channels of a standard digital image into distinct data planes. In this morphological framework, an image is not merely a static graphic but a      
    dense instruction grid where each pixel, or "codel," carries specific programmatic weight.2 This method of encoding draws conceptual inspiration    
    from earlier esoteric languages while significantly professionalizing the execution model for modern Graphics Processing Units (GPUs).6             
                                                                                                                                                        
    In this specific encoding scheme, the Red (R) channel is dedicated to the visual structure and structural integrity of the glyphs, ensuring that    
    the program remains recognizable to human developers and visual recognition systems. The Green (G) channel serves as the primary instruction        
    carrier, with each intensity value mapping directly to a specific GeoASM opcode. The Blue (B) channel acts as the operand and register metadata     
    store, providing the context required for the execution of the Green channel's logic \[geometry-os-spirv skill\].                                   
                                                                                                                                                        
    ### **Channel Functionality and Hardware Mapping**

    identifies an addition operation, which the emitter subsequently translates into the OpFAdd SPIR-V instruction.8 Similarly, the Blue channel        
    dictates the data flow, where values ![][image2] through ![][image3] identify GPU registers ![][image4] through ![][image5], while values from

    ## **The Mathematics of Locality: Hilbert Curve Instruction Sequencing**                                                                            
                                                                                                                                                        
    A fundamental challenge in visual-to-bytecode compilation is the discrepancy between the two-dimensional arrangement of visual glyphs and the       
    linear execution model of the hardware. Traditional linear scanning methods, such as raster scanning, are inherently flawed when applied to visual  
    logic because they fail to preserve functional proximity. When logic is designed in a cluster, a raster scan will break the connection between      
    vertically adjacent glyphs, leading to fragmented instruction streams and poor cache performance.3                                                  
                                                                                                                                                        
    To address this, Geometry OS employs the Hilbert curve to reorder the 2D visual layout into a 1D instruction stream.13 The Hilbert curve is a       
    fractal, space-filling curve that maps a one-dimensional interval onto a higher-dimensional square while maintaining a high degree of spatial       
    locality.14 This ensures that glyphs that are close together in the 2D grid—and therefore likely related in logic—remain close together in the      
    serialized bytecode passed to the GPU.13                                                                                                            
                                                                                                                                                        
    ### **Implementation via Skilling’s Algorithm**                                                                                                     
                                                                                                                                                        
    The conversion between two-dimensional coordinates ![][image10] and a one-dimensional Hilbert index is typically managed via Skilling’s Algorithm.  
    Unlike recursive methods that can be computationally expensive, Skilling’s method relies on bit-level manipulation, Gray code transformations, and  
    XOR operations to calculate positions.17 The algorithm operates on the principle that a grid of ![][image11] cells can be addressed by ![][image12] 
     bits per axis.4                                                                                                                                    
                                                                                                                                                        
    The essential steps for mapping a Hilbert index back to a set of spatial coordinates involve:                                                       
                                                                                                                                                        
    1. Converting the Hilbert index from decimal to its binary representation.                                                                          
    2. Transforming the binary value into its corresponding Gray code.                                                                                  
    3. Disentangling the Gray code structure into individual bits for the X and Y coordinates.                                                          
    4. Applying iterative rotations and reflections based on the quadrant of the sub-curve.17                                                           
                                                                                                                                                        
    Mathematically, the Hilbert curve ![][image13] for a grid of size ![][image14] is defined by its recursive structure, but for the purposes of       
    high-speed compilation, the bit-twiddling approach is preferred for its ![][image15] efficiency where ![][image16] is the number of bits.18 By      
    preserving locality, the Hilbert curve enables the compiler to better manage Static Single Assignment (SSA) identifiers, as the dependencies        
    between instructions are more likely to be contiguous in the stream.1                                                                               
                                                                                                                                                        
    ## **SPIR-V Binary Module Internals and Structure**                                                                                                 
                                                                                                                                                        
    The target of the Geometry OS pipeline is a binary SPIR-V module, a strictly formatted unit of GPU bytecode that represents a control-flow graph in 
     a portable format.22 SPIR-V is designed to be regular and predictable, facilitating easier ingestion by drivers from diverse hardware vendors.1 A  
    valid SPIR-V module is comprised of a sequence of 32-bit words, beginning with a mandatory five-word header that establishes the environment for    
    the subsequent instructions.25                                                                                                                      
                                                                                                                                                        
    ### **The 5-Word Header Specification**                                                                                                             
                                                                                                                                                        
    The header is the first element of any .spv file and provides critical metadata for the GPU driver.25                                               
                                                                                                                                                        
    | Word | Field Name | Typical Value | Description |                                                                                                 
    | :---- | :---- | :---- | :---- |                                                                                                                   
    | 0 | Magic Number | 0x07230203 | Identifies the file as SPIR-V and determines endianness.25 |                                                      
    | 1 | Version | 0x00010000 | The specification version (e.g., 1.0).25 |                                                                             
    | 2 | Generator | e.g., 22 | A unique ID for the tool that generated the module.26 |                                                                
    | 3 | ID Bound | e.g., 14 | The maximum value of any SSA result ID used in the module plus one.25 |                                                 
    | 4 | Schema | 0 | Reserved for future use; currently always set to zero.25 |                                                                       
                                                                                                                                                        
    The "Magic Number" is particularly significant for cross-platform compatibility. If a system reads the header and finds 0x03022307, it indicates    
    that the file was generated with a different endianness, allowing the driver to swap every 4-byte word in a simple loop without needing to parse    
    the individual instructions first.24                                                                                                                
                                                                                                                                                        
    ### **Logical Ordering of Sections**                                                                                                                
                                                                                                                                                        
    Following the header, the SPIR-V specification requires instructions to follow a rigid logical order. This order ensures that dependencies are      
    satisfied before an ID is used. For instance, a capability must be declared before any instruction that requires it can be parsed.1                 
                                                                                                                                                        
    1. **Capabilities**: Declared via OpCapability (e.g., Shader, Matrix).                                                                              
    2. **Extensions**: Declared via OpExtension.                                                                                                        
    3. **Extended Instruction Sets**: Imported via OpExtInstImport (e.g., GLSL.std.450).                                                                
    4. **Memory Model**: Defined via OpMemoryModel (usually Logical GLSL450).                                                                           
    5. **Entry Points**: Declared via OpEntryPoint, which links a specific function to a shader stage like Fragment or GLCompute.25                     
    6. **Execution Modes**: Declared via OpExecutionMode (e.g., OriginUpperLeft).                                                                       
    7. **Debug Information**: Grouped instructions like OpName and OpSource for debugging.                                                              
    8. **Annotations**: Metadata attached to IDs via OpDecorate (e.g., Location for interface variables).23                                             
    9. **Types, Constants, and Variables**: Declarations of scalar types, vector types, and global constants.22                                         
    10. **Functions**: The procedural logic of the kernel, beginning with OpFunction and ending with OpFunctionEnd.23                                   
                                                                                                                                                        
    Each instruction in this stream is represented as a single 32-bit word containing the word count in the upper 16 bits and the opcode in the lower   
    16 bits, followed by the operands.25 This structure allows tools to skip over unknown or unsupported instructions by simply jumping the number of   
    words specified in the instruction header.1                                                                                                         
                                                                                                                                                        
    ## **Direct Emission Methodology via Python**                                                                                                       
                                                                                                                                                        
    The translation of Geometry OS glyphs into binary bytecode is performed through the direct emission of 32-bit words using Python’s struct module.   
    This approach provides the flexibility required to generate valid modules without the overhead of heavy compiler frameworks.31 The struct.pack      
    function is used to convert Python integers and floats into C-style binary data, typically using the \<I format string to ensure little-endian      
    32-bit unsigned integers.33                                                                                                                         
                                                                                                                                                        
    ### **Constructing Instructions and ID Management**                                                                                                 
                                                                                                                                                        
    Because SPIR-V utilizes Static Single Assignment (SSA) form, every instruction that generates a result must produce a unique ID.1 The Geometry OS   
    compiler maintains a global counter to assign these IDs and tracks the "ID Bound" to update the module header accordingly.25 For a complex visual   
    program, this requires a robust mapping between the virtual registers identified in the Blue channel and the physical SSA IDs allocated during      
    emission.20                                                                                                                                         
                                                                                                                                                        
    When the emitter encounters a glyph representing a floating-point addition (0x6A in the G-channel), it constructs the binary word sequence for      
    OpFAdd (opcode ![][image17]). The sequence would typically look like this:                                                                          
                                                                                                                                                        
    1. Instruction header: (5 \<\< 16\) | 129 (a 5-word instruction with opcode 129).                                                                   
    2. Result Type ID: The ID of the OpTypeFloat 32 declaration.                                                                                        
    3. Result ID: The unique ID assigned to this operation's output.                                                                                    
    4. Operand 1: The ID of the first value being added.                                                                                                
    5. Operand 2: The ID of the second value being added.25                                                                                             
                                                                                                                                                        
    Floating-point literals are packed using the f format character, which follows the IEEE 754 binary32 standard, ensuring that the GPU interprets the 
     constant correctly regardless of the host's native floating-point format.32                                                                        
                                                                                                                                                        
    ### **Optimization and Validation**                                                                                                                 
                                                                                                                                                        
    While direct emission is efficient, the resulting bytecode must be validated to ensure it adheres to the Khronos universal limits and logical       
    constraints. The Vulkan SDK provides spirv-val, which performs three levels of validation, checking for type mismatches, invalid SSA form, and      
    section order violations.34 Furthermore, tools like spirv-opt can be applied to the resulting binary to remove redundant OpName instructions,       
    collapse dead code, and canonicalize IDs for better performance.35                                                                                  
                                                                                                                                                        
    ## **Prior Art: From Piet to Geometry OS**                                                                                                          
                                                                                                                                                        
    The concept of encoding logic within color and spatial structures has deep roots in the field of esoteric programming languages. The primary        
    inspiration for Geometry OS is Piet, a language where programs are bitmap images that look like abstract geometric art.2                            
                                                                                                                                                        
    ### **Piet and the Mechanics of Hue-Lightness Transitions**                                                                                         
                                                                                                                                                        
    Named after Piet Mondrian, the language Piet uses 20 distinct colors arranged in cycles of hue and lightness.5 Unlike Geometry OS, which uses fixed 
     opcode mapping, Piet triggers commands based on the relative shift between two adjacent color blocks.2 For example, a shift from red to yellow     
    might represent addition if they share the same lightness level.2                                                                                   
                                                                                                                                                        
    | Shift Type | Effect in Piet | Geometry OS Counterpart |                                                                                           
    | :---- | :---- | :---- |                                                                                                                           
    | Hue Shift | Determines the core operation (Add, Sub, etc.) | Green Channel Value |                                                                
    | Lightness Shift | Determines the specific variant of the operation | Constant mapping within opcode set |                                         
    | Block Size | Pushes a value onto the stack | Blue Channel Immediate Value |                                                                       
                                                                                                                                                        
    Piet is stack-based and relies on a "Direction Pointer" (DP) and a "Codel Chooser" (CC) to navigate the program maze.5 While highly expressive and  
    artistic, Piet is notoriously difficult to debug and logic is easily broken by minor pixel adjustments.7 Geometry OS solves these problems by       
    moving from a relative transition model to an absolute mapping model, making the language suitable for the deterministic requirements of GPU        
    compute kernels \[geometry-os-spirv skill\].                                                                                                        
                                                                                                                                                        
    ### **Befunge and Dimensional Roaming**                                                                                                             
                                                                                                                                                        
    Befunge is another significant ancestor, being the first widely known language to arrange code on a two-dimensional grid where the instruction      
    pointer can roam in multiple directions.38 While Befunge remains a textual language (using ASCII characters), its emphasis on spatial logic and     
    grid-based control flow provided the foundational theory for how visual programs can be structured in non-linear ways.39 Geometry OS takes the      
    dimensional roaming of Befunge and the chromatic encoding of Piet and fuses them into a hardware-aware pipeline targeting SPIR-V.5                  
                                                                                                                                                        
    ## **Technical Integration with Vulkan and WebGPU**                                                                                                 
                                                                                                                                                        
    Once the visual pattern has been compiled into a SPIR-V binary, it can be executed on a variety of platforms via modern graphics APIs. The          
    integration with Vulkan is the most direct, as Vulkan accepts raw SPIR-V bytecode as its native shader format.40                                    
                                                                                                                                                        
    ### **Loading Shaders in Vulkan**                                                                                                                   
                                                                                                                                                        
    To feed the generated bytecode into Vulkan, the application uses the vkCreateShaderModule function. This requires the bytecode to be provided as a  
    pointer to an array of 32-bit unsigned integers.27 On Windows systems, it is critical that the .spv file is opened in "rb" (binary read) mode. If   
    opened in text mode, the system may interpret the byte 0x1A as an end-of-file (EOF) marker, leading to truncated shader code and module creation

    }

    ### **WebGPU and the WGSL Constraint**                                                                                                              
                                                                                                                                                        
    Integration with WebGPU is more complex due to the standard's focus on a text-based intermediate language, WGSL.44 While some implementations of    
    WebGPU (like those based on the wgpu crate in Rust) can accept SPIR-V as an input by translating it to WGSL at runtime, other backends like Dawn    
    may require explicit WGSL descriptors.45 For the Geometry OS pipeline to remain truly portable, it must utilize tools like SPIRV-Cross to decompile

    ## **Systemic Insights: Spatial Logic as a Computational Strategy**                                                                                 
                                                                                                                                                        
    The shift from textual to morphological programming in Geometry OS reveals several deep systemic advantages that extend beyond the simple act of    
    writing code. By treating a program as a 2D geometric structure, we can apply techniques from image processing and spatial analysis to code         
    optimization.                                                                                                                                       
                                                                                                                                                        
    ### **Cache Coalescing and the "Program as Texture"**                                                                                               
                                                                                                                                                        
    One of the most profound insights of using the Hilbert curve for instruction reordering is its effect on the compiler's memory access patterns.     
    Because the Hilbert curve visits spatially adjacent glyphs in a continuous sequence, the extraction of opcodes and operands from the image becomes  
    a highly coalesced operation.12 This mirrors the ways GPUs optimize texture sampling using Morton (Z-order) curves to ensure that texels near each  
    other in 2D space occupy the same cache line in 1D memory.12 In Geometry OS, the kernel is essentially a "code texture," and the compilation        
    process is optimized by the same mathematical principles that make GPU graphics fast.48                                                             
                                                                                                                                                        
    ### **The SSA Form as a Spatio-Temporal Puzzle**                                                                                                    
                                                                                                                                                        
    The requirement of Static Single Assignment (SSA) in SPIR-V introduces a temporal constraint on the 2D visual grid. In an SSA-form program, every   
    variable must be defined exactly once before it can be used.1 This means that the path of the Hilbert curve creates a "causal timeline" across the  
    grid. A glyph placed at coordinate ![][image10] can only use operands defined by glyphs that appear *earlier* in the Hilbert sequence.1             
                                                                                                                                                        
    This suggests that the layout of a visual program is not just a matter of aesthetics, but a puzzle of topological arrangement. If a programmer      
    creates a dependency loop in 2D space that cannot be resolved by the linear Hilbert traversal, the resulting SPIR-V module will be invalid.34 This  
    forces the developer to think in terms of "logical flow" through geometry, where the 2D arrangement must respect the 1D serialization order         
    required by the hardware.                                                                                                                           
                                                                                                                                                        
    ### **Multi-Modal Semantic Expressiveness**                                                                                                         
                                                                                                                                                        
    The use of distinct color channels for different programmatic layers enables a form of "semantic communications" where the program's data is more   
    difficult to reverse-engineer than traditional text.49 Because the functional information (G and B channels) is realigned into a unified semantic   
    space, it is possible to create "steganographic code" where a functional compute kernel is hidden within a visually innocuous image.50 This         
    layering of visual "marks" and functional "channels" aligns with established principles of visual encoding, where the most important attributes are 
     matched to the most effective channels.11                                                                                                          
                                                                                                                                                        
    | Channel | Logical Role | Encoding Type | Effectiveness Bias |                                                                                     
    | :---- | :---- | :---- | :---- |                                                                                                                   
    | Red | Appearance / UI | Iconic / Inferred | High-focus / Semantic 11 |                                                                            
    | Green | Execution Logic | Deterministic / Encoded | Low-bias / Logical |                                                                          
    | Blue | Context / Data | Meta-data / Register | Saturated-is-more 52 |                                                                             
                                                                                                                                                        
    By leveraging these biases, Geometry OS provides a platform where complex 3D geometry information can be extracted from 2D depth maps and used as   
    priors for allocating attention weights in self-attention mechanisms, effectively using the program’s visual input as part of its internal logic.53

      Now that we have a stable, documented pipeline with 8 opcodes and a Hilbert visualizer, we can move toward making this a "Real OS" environment. I 
      recommend we tackle Memory Management next.                                                                                                       
                                                                                                                                                        
                                                                                                                                                        
      Why Memory Management?                                                                                                                            
      Currently, programs are purely transient (stack-only). Adding load and store operations will allow:                                               
       * Variables: Assigning results to specific "RAM" slots.                                                                                          
       * State: Programs that can remember values across different execution blocks.                                                                    
       * Foundational OS Logic: Managing a heap or registry of data.                                                                                    
                                                                                                                                                        
                                                                                                                                                        
      Alternatively, we could do Control Flow:                                                                                                          
       * Conditionals: Using > and < to jump over certain instructions.                                                                                 
       * Loops: Repeating a Hilbert sequence based on a counter.

    Decision Matrix
  
    ┌────────────────┬────────────────────────────────────────┬─────────────────────────────────────────┐
    │     Factor     │           Memory Management            │              Control Flow               │
    ├────────────────┼────────────────────────────────────────┼─────────────────────────────────────────┤

    ├────────────────┼────────────────────────────────────────┼─────────────────────────────────────────┤

    ├────────────────┼────────────────────────────────────────┼─────────────────────────────────────────┤
    │ Hilbert impact │ Minimal (linear access)                │ Significant (jumps break linear flow)   │
    └────────────────┴────────────────────────────────────────┴─────────────────────────────────────────┘
  
    My Recommendation: Memory Management First
  
    Why?
  
    1. Natural fit for Hilbert model - Load/store doesn't break the linear execution flow
    2. Enables Control Flow - Loops need counters, conditionals need comparison storage
    3. Simpler implementation - Just add load and store opcodes + a RAM buffer
    4. Immediate utility - You can already do useful computation, now you can save results
