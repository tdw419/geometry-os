  # Geometry OS: Visual Computing Architecture
  
  This project is a **Visual Operating System substrate**. It replaces the traditional text-based "symbolic" computing model with a "geometric" model where the screen is the hard drive and pixels are the instructions.
  
  ### 1. The Current Implementation (Phase 20-25)
  
  | Component | Technology | Purpose |
  | :--- | :--- | :--- |
  | **Visual Substrate** | `.rts.png` | Programs stored as images with Hilbert-curve locality. |
  | **Compiler** | `visual_to_spirv.py` | PNG (Semantic RGB) → SPIR-V Bytecode. |
  | **Execution** | `executor.wgsl` | GPU-side Virtual Machine (Stack-based) running in WebGPU. |
  | **Interface** | `VisualIDE.js` | Interactive grid-based editor for "drawing" code. |
  
  ### 2. How to Build an OS from This
  
  To transition from a "toolkit" to a full "Operating System" like Linux or Windows, we are following the **Geometry OS Roadmap**:
  
  1.  **Memory Management (Next Phase):** Implement "Visual Pages" where the GPU can swap instruction textures (PNGs) in and out of the active execution buffer.
  2.  **Process Isolation:** Use WebGPU `RenderGroups` to isolate different visual programs on the same infinite map.
  3.  **Hardware Abstraction:** Treat the GPU as the primary CPU. Disk I/O becomes "Pixel I/O" (saving/loading textures).
  4.  **Infinite Map Compositor:** The `systems/infinite_map_rs` (Rust/Smithay) component is the "Kernel" that manages the spatial windowing of these visual programs.
